We need to harden the image-loading fix for our React + Vite + Express + Capacitor iOS marketplace app.

Context:
- App is React 18 + Vite + Express TypeScript, deployed on Replit.
- Capacitor iOS wrapper loads the live app via server.url: https://thesamanapp.com.
- Product images are stored in Google Cloud Storage.
- DB stores permanent paths like /objects/uploads/<uuid>, not signed URLs.
- Server endpoint GET /objects/(.*) generates a fresh signed GCS URL and returns a 302 redirect.
- Previous bug: iOS WKWebView had stale cached signed GCS URLs from an old expiry-window issue. Some older images broke only on one physical iPhone, while fresh devices/incognito worked fine.
- We already added cache busting with ?_=v3 and a retryObjectImg onError helper.
- The current retry approach is valid, but I want it polished and the redirect cache tightened.

Please make these changes:

1. Improve the retryObjectImg helper in:
client/src/lib/bustObjectUrl.ts

Current idea:
- Keep bustObjectUrl behavior: only add ?_=v3 to URLs starting with /objects/ if they do not already contain _=.
- Improve retryObjectImg so it:
  - Retries only once using data-retried="1".
  - Only acts on image src values that contain /objects/.
  - Uses URL + searchParams instead of src.split("?")[0], so future query params are preserved.
  - Sets _=r<Date.now()> to force a unique cache key.
  - Has a safe fallback using split("?")[0] if URL parsing fails.

Target implementation:

export function retryObjectImg(e: React.SyntheticEvent<HTMLImageElement>) {
  const img = e.currentTarget;

  if (img.dataset.retried === "1") return;

  const src = img.getAttribute("src") || "";
  if (!src.includes("/objects/")) return;

  img.dataset.retried = "1";

  try {
    const url = new URL(src, window.location.origin);
    url.searchParams.set("_", `r${Date.now()}`);
    img.src = url.pathname + url.search;
  } catch {
    const base = src.split("?")[0];
    img.src = `${base}?_=r${Date.now()}`;
  }
}

3. Tighten the /objects/* redirect Cache-Control header in the Express server.

Find the route:
GET /objects/(.*)

It currently generates a signed GCS URL and returns a 302 redirect.

Change the redirect response Cache-Control from:
Cache-Control: public, max-age=72000, s-maxage=72000

to:
Cache-Control: public, max-age=300, s-maxage=300

Reason:
- The signed GCS URL lasts around 5–6 days, but mobile WebViews can behave badly with stale cached redirects/final signed URLs.
- A 5-minute redirect cache still helps performance but dramatically reduces the blast radius if a stale or bad redirect ever gets cached.
- Do not use no-store unless necessary; 300 seconds is the preferred balance.
